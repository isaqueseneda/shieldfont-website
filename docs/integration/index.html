Integrating ShieldFont — the four tiers

ShieldFont ships in four flavors depending on how you build pages. Pick the one that matches your stack:

TierAudienceInstallEncoding happens...
A. JSX (<Shield> component)React / Next.js / Astro / Remixnpm i @shieldfont/reactAt server-render time
B. HTML build scriptStatic-site builders, CI pipelinesnpm i -D @shieldfont/cliAt shieldfont build time
C. Copy-paste embedNaive HTML devs, "I just want a snippet"none — pre-baked on s-a.websitePre-encoded server-side
D. Downloadable fontMicrosoft Word, Pages, InDesign, PDF authorsone-click .zipAt Word/PDF render time via OpenType ligatures

All four tiers share the same M15-EN mapping — just different delivery mechanisms.

The architectural rule we won't cross

All encoding happens at build time. Never in the browser. Never at HTTP-response time.

  • Browser-runtime JS encoders are fundamentally broken: scrapers don't run JS, so they see your plain English. Protection is moot.
  • Edge-middleware encoding is technically secure but adds runtime cost on every request. We're staying out of that space.

The encoded form is what's stored, what's served, what's cached. Identical to how Tailwind compiles classes at build time.

Dynamic sites

If you're building a React / Next.js / Remix / Astro app, you ship Shieldfont as a server component. Encoding happens at SSR — no runtime cost in the browser, no build script. See Tier A below for the full integration.

Jump to Tier A — JSX with @shieldfont/react ↓

Tier A — JSX with @shieldfont/react

The recommended path for vibe-coders, Next.js apps, Astro, Remix, and any React Server Component framework.

Install

npm install @shieldfont/react

Use

import { Shield } from "@shieldfont/react";

export default function Page() {
  return (
    <article>
      <h1>About us</h1>             {/* not protected — plain font */}

      <Shield>
        The future of writing belongs to those who protect their words.
      </Shield>

      <Shield as="p" weight={500} lineHeight={1.7}>
        Our mission is to build a publishing layer that the open web can trust.
      </Shield>

      <Shield as="h2" size="2.4rem">
        Manifesto
      </Shield>
    </article>
  );
}

That's it. The font + @font-face + encoding all happen automatically. Anything outside <Shield> uses your normal page fonts.

Self-host the font (optional)

By default, the bundled font is served from jsDelivr (free public CDN, no setup needed). To self-host:

// somewhere in your app's bootstrap (server-side)
import { setFontHost } from "@shieldfont/react";
setFontHost("/fonts");  // and copy the woff2/ttf into public/fonts/

The bundled font files live at node_modules/@shieldfont/react/fonts/shieldfont-optik-en-m15-alpha.{woff2,ttf}.

Verify it's working

After running next build (or astro build or whatever), scrape your own page:

curl https://your-site.com/some-protected-page | grep 'data-shieldfont'

You should see encoded text in the HTML, not the original English. Open the same URL in a browser — humans see the original because the font reverses the encoding visually.

Restrictions

  • Children must be a string. No nested JSX. For mixed content (text + links), split into multiple <Shield> instances.
  • Server-only. v1 has no client component — encoding must happen on the server so the encoded text reaches the browser.

Blogs and static sites

If your site is plain HTML, an SSG (Astro / 11ty / Hugo / Jekyll), or any blog where you can drop in a script, you use the HTML embed flow. You write plain English in your source, wrap protected blocks with comment markers, and run shieldfont build as part of your CI. See Tier B below for the full integration.

Jump to Tier B — HTML + CLI ↓

Tier B — HTML + CLI (@shieldfont/cli)

For static-site generators (Astro, 11ty, Hugo), CI pipelines, or any HTML build flow.

Install

npm install --save-dev @shieldfont/cli
# Or use without installing:
npx shieldfont --help

The comment-marker convention

Mark protected regions in your HTML with comment markers. Two flavors:

First-time wrapping (you write this):

<!-- shield-on -->
<h1>The future of writing</h1>
<p>Belongs to those who write it.</p>
<!-- shield-off -->

Run npx shieldfont build path/to/file.html once. The CLI normalizes that block into per-text-node markers:

After build (the file in your repo now looks like this):

<h1><!-- shield: The future of writing -->The future for watching<!-- /shield --></h1>
<p><!-- shield: Belongs to those who write it. -->Belongs to these who write him.<!-- /shield --></p>

The opening comment carries the source-of-truth (plain English). The text between the markers is what's displayed (encoded). To edit copy, change the text inside the comment and re-run build — the visible text regenerates idempotently.

Before deploying: run shieldfont ship to strip all comments so the deployed HTML is camouflage-clean (zero on-page signal that ShieldFont was used):

<!-- production HTML, after ship -->
<h1>The future for watching</h1>
<p>Belongs to these who write him.</p>

Commands

shieldfont encode "text"           # one-off encoding to stdout
shieldfont decode "text"           # reverse (mapping is bidirectional)
shieldfont build  [glob...]        # re-derive visible text from <!-- shield: ... --> markers (idempotent)
shieldfont ship   [glob...]        # strip all shield-related comments
shieldfont check  [glob...]        # verify markers round-trip; non-zero exit on mismatch

Globs default to **/*.html when no path is given. Build/ship/check edit files in place — use git or make a backup.

Recommended build pipeline

In your package.json:

{
  "scripts": {
    "dev": "shieldfont build src/**/*.html",
    "build": "your-static-site-generator && shieldfont build dist/**/*.html",
    "ship": "shieldfont check dist/**/*.html && shieldfont ship dist/**/*.html"
  }
}

Loading the font

Since @shieldfont/cli doesn't touch your CSS, you load the font yourself. The simplest path is the same CDN the JSX tier uses:

<link rel="stylesheet" href="https://cdn.jsdelivr.net/gh/isaqueseneda/shieldfont@v2.2.0/packages/react/fonts/shieldfont-optik-en-m15-alpha.css">
<style>
  [data-shieldfont] { font-family: 'ShieldFont Optik EN M15 Alpha', sans-serif; }
</style>

Then add data-shieldfont to elements that should use the protected font scope.

Tier C — Copy-paste embed

The lowest-friction path for non-developers, blog authors, or "I just want a snippet to paste."

Visit s-a.website/shieldfont/embed (coming v2.2):

  1. Paste your plain English text
  2. Pick presets: heading-size, body-size, weight, line-height, color
  3. Click "Generate"
  4. Copy the snippet, paste into your HTML

The generated snippet is fully self-contained — single <link> to a CDN-hosted font + a <div> with the encoded text inline. No install, no CLI, no JSX.

Tier D — Download (Microsoft Word / PDF)

For journalists, document authors, anyone sending PDFs through email.

Download the .zip from s-a.website/shieldfont/download (coming v2.2):

shieldfont-optik-en-m15-alpha.zip
  shieldfont-optik-en-m15-alpha.ttf       <- install in Word, Pages, InDesign
  shieldfont-optik-en-m15-alpha.otf       <- compatibility with older apps
  shieldfont-optik-en-m15-alpha.woff2     <- web (in case)
  README.md                                <- 1-page how-to
  encoder.html                             <- standalone offline encoder you open in any browser

Workflow:

  1. Install the .ttf or .otf on your system.
  2. Open Word / Pages / InDesign. Set the font for paragraphs you want to protect.
  3. Type normally. The font's GSUB ligatures encode the text visually as you type — you see plain English; the underlying glyphs are encoded.
  4. Export to PDF. The encoded form is what's stored in the PDF.
  5. Email the PDF. If your recipient's email provider scrapes attachments to train AI models, the encoded text is useless training data.

For documents you'll edit later, also keep a plain-English source copy somewhere (the encoder.html in the zip lets you re-encode any time).

Versioning

Every CDN URL we publish is version-pinned and immutable. No "latest" channels — silently upgrading the mapping would break existing encoded content.

✅ https://cdn.jsdelivr.net/gh/isaqueseneda/shieldfont@v2.2.0/packages/react/fonts/...
❌ https://cdn.jsdelivr.net/gh/isaqueseneda/shieldfont@latest/...

Upgrading to a new mapping version is opt-in: re-run shieldfont build (Tier B), bump the npm package (Tier A), regenerate your snippet (Tier C), or re-download (Tier D).

Threat model — what ShieldFont does and doesn't protect against

Defends:

  • Naive HTML scrapers (curl + regex, requests + BeautifulSoup, trafilatura, readability-lxml)
  • Bulk dataset pipelines that read innerText without rendering fonts
  • Email attachment scrapers (PDF/DOCX exports keep encoded text)
  • Copy-paste into text-only tools

Does not defend:

  • Headless browsers that fully render fonts (Playwright, Puppeteer)
  • OCR on rendered pages or screenshots
  • Vision-language models reading screenshots
  • Frequency analysis on a large corpus (per-deploy random seeds coming in v2.3)

Our framing: ShieldFont raises the cost of extraction; it doesn't promise zero extraction.
